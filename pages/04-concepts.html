第 4 章 · 学 Paimon

核心概念

这一章把 Paimon 文档里最常出现的术语，用通俗的话讲一遍。看懂这些，文档就不再像天书。

先建立一个直觉

Paimon 处理「更新」的方式和你想的不太一样。它不是直接去改旧文件，而是写一份新数据 + 记录一次元数据变化，再用「版本」来决定哪些数据对外可见。下面这些概念都是围绕这个思路展开的。

关键术语速查

术语通俗解释
snapshot快照。某个时间点的「数据版本」，读表时从一个 snapshot 出发还原出完整视图。
manifest清单文件。记录这个版本里有哪些数据文件被「加」或「删」。
data file真正存数据的文件（列式存储，如 Parquet/ORC）。
LSM tree一种「先写内存、再落盘成有序块、后台再合并」的结构，让高频更新很高效。
compaction压缩合并。把零散的小文件 / 多层数据整理合并，提升查询性能。
changelog变更日志。把数据的增删改作为「变化流」输出，供下游增量计算。
time travel时间旅行。查询历史某个 snapshot 的数据。
schema evolution表结构演进。加字段、改类型时不用重写历史数据。
merge engine合并引擎。决定主键相同的多条记录怎么合并（去重、部分更新、聚合等）。

三件最容易踩坑的事

① 删除不等于立刻删文件

在 Paimon 里，删除往往先是「逻辑删除」——对用户不可见，但文件还在。真正的物理删除要等 snapshot 过期（expiration） 才发生。

② 读表靠元数据，不是扫目录

Paimon 读表是从 snapshot → manifest → data file 这棵元数据树出发的，而不是简单看目录里有哪些文件。

③ 小文件和 checkpoint 有关

流式写入时每个 Flink checkpoint 会生成快照，checkpoint 间隔越小，小文件越多。需要靠 compaction、bucket 设置等来治理。

主键表为什么适合更新

主键表用 LSM 思路：新数据先进内存缓冲，再落盘成有序块（sorted run），后台异步做 compaction 合并。这样高频 upsert（有就更新、没有就插入）才不会每次都重写整张表，性能才扛得住。
